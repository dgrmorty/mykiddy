# Откуда брать значения переменных окружения — пошагово

Ниже для каждой переменной: **где взять** и **что подставить**.

---

## Серверные переменные (для `node server.js` / Railway)

### PORT
**Что это:** порт, на котором крутится сервер.

**Где взять:** нигде — это число ты задаёшь сам.

**Что подставить:**
- Локально: можно не задавать, по умолчанию будет `3000`.
- На Railway: обычно **не задаёшь** — Railway сам подставляет порт через переменную `PORT`; если в настройках сервиса уже есть `PORT`, не трогай.

---

### API_KEY
**Что это:** ключ Google Gemini API для ИИ-тьютора и проверки ДЗ.

**Где взять:**
1. Открой в браузере: **https://aistudio.google.com/apikey**
2. Войди в Google-аккаунт.
3. Нажми **«Create API key»** (или «Get API key»).
4. Выбери проект или создай новый → скопируй ключ (длинная строка вида `AIza...`).

**Что подставить:** вставил скопированный ключ в значение `API_KEY` в `.env` или в Variables в Railway.  
Пример в `.env`: `API_KEY=AIzaSyB...твой_ключ...`

---

### SUPABASE_URL
**Что это:** адрес твоего проекта Supabase.

**Где взять:**
1. Зайди на **https://supabase.com** и войди в аккаунт.
2. Открой свой проект (или создай новый).
3. Слева меню: **Project Settings** (иконка шестерёнки).
4. Вкладка **API**.
5. Блок **Project URL** — там ссылка вида `https://xxxxx.supabase.co`.

**Что подставить:** эту ссылку целиком, без слеша в конце.  
Пример: `SUPABASE_URL=https://twaepaurydscpcgfgtuc.supabase.co`

---

### SUPABASE_ANON_KEY
**Что это:** публичный (anon) ключ проекта Supabase. Им пользуются и фронт, и сервер (доступ ограничивается RLS).

**Где взять:**
1. Тот же раздел: **Project Settings → API**.
2. Блок **Project API keys**.
3. Строка **anon** **public** — нажми **Reveal** и скопируй ключ (длинный JWT).

**Что подставить:** скопированную строку.  
Пример: `SUPABASE_ANON_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...`

---

### CORS_ORIGIN (необязательно)
**Что это:** доп. разрешённые домены для запросов к API (если фронт открыт с другого домена).

**Где взять:** нигде — это твой домен.

**Что подставить:** только если фронт и API на **разных** доменах. Укажи URL фронта без слеша в конце.  
Пример: `CORS_ORIGIN=https://mykiddy.ru`  
Если всё на одном домене (например, только Railway) — переменную можно не задавать.

---

## Клиентские переменные (для сборки фронта / Vite)

Они подставляются **в момент сборки** (`npm run build`). В Railway их нужно добавить в **Variables** сервиса, который собирает фронт, чтобы при деплое они уже были.

### VITE_API_URL
**Что это:** адрес твоего бэкенда (API), к которому ходит фронт.

**Где взять:** это URL твоего приложения.

**Что подставить:**
- **Локально:** можно не задавать (будет использоваться текущий origin или proxy из Vite).
- **Продакшен (Railway):** после деплоя в Railway открой свой сервис — там будет ссылка вида `https://mykiddy-production.up.railway.app`. Эту ссылку и поставь.  
  Пример: `VITE_API_URL=https://mykiddy-production.up.railway.app`  
  Без слеша в конце.

---

### VITE_SUPABASE_URL
**Что это:** тот же адрес проекта Supabase, что и для сервера.

**Где взять:** как для **SUPABASE_URL** выше — **Supabase → Project Settings → API → Project URL**.

**Что подставить:** то же значение, что и в `SUPABASE_URL`.  
Пример: `VITE_SUPABASE_URL=https://twaepaurydscpcgfgtuc.supabase.co`

---

### VITE_SUPABASE_ANON_KEY
**Что это:** тот же anon-ключ, что и для сервера.

**Где взять:** как для **SUPABASE_ANON_KEY** выше — **Supabase → Project Settings → API → anon public**.

**Что подставить:** то же значение, что и в `SUPABASE_ANON_KEY`.  
Пример: `VITE_SUPABASE_ANON_KEY=eyJhbGciOiJIUzI1NiIs...`

---

### VITE_ADMIN_EMAILS
**Что это:** список email-адресов, которые считаются админами (доступ в панель управления).

**Где взять:** нигде — это твой (или твоих админов) email.

**Что подставить:** один или несколько email через запятую, без пробелов. Тот аккаунт, которым ты входишь в приложение и хочешь видеть раздел «Управление», должен быть в этом списке.  
Пример: `VITE_ADMIN_EMAILS=knazar002@gmail.com`  
Несколько: `VITE_ADMIN_EMAILS=admin@school.ru,teacher@school.ru`

---

## Где это всё прописать

### Локально (у себя на компе)
1. В корне проекта создай файл **`.env`** (рядом с `.env.example`).
2. Скопируй туда строки из `.env.example` и подставь значения по инструкции выше.
3. Переменные с префиксом `VITE_` будут подхвачены при `npm run dev` и `npm run build`.

### На Railway
1. Открой проект в **https://railway.app** → свой сервис.
2. Вкладка **Variables** (или **Settings → Variables**).
3. Добавь каждую переменную: имя (например `API_KEY`) и значение (вставь ключ/URL).
4. Для **VITE_*** важно: они должны быть заданы **до сборки**. Если сборка идёт в Railway при деплое — добавь их в Variables этого же сервиса; тогда при `npm run build` они подставятся в фронт.
5. Сохрани; при необходимости сделай **Redeploy**.

## Минимальный набор для старта

Чтобы всё заработало, обязательно нужны:

| Переменная            | Где взять кратко                    |
|-----------------------|-------------------------------------|
| **API_KEY**           | https://aistudio.google.com/apikey  |
| **SUPABASE_URL**      | Supabase → Project Settings → API → Project URL |
| **SUPABASE_ANON_KEY** | Supabase → Project Settings → API → anon public |
| **VITE_SUPABASE_URL** | То же, что SUPABASE_URL             |
| **VITE_SUPABASE_ANON_KEY** | То же, что SUPABASE_ANON_KEY   |
| **VITE_ADMIN_EMAILS** | Твой email для входа в админку     |

**PORT** и **VITE_API_URL** часто можно не задавать (подставляются по умолчанию или из окружения Railway).

После того как заполнишь `.env` по этой инструкции, можно прислать только **список имён** переменных (без значений), и я проверю, что ничего не забыто.

## Supabase CLI и продакшн

Проект уже **связан** с облачным проектом (после `supabase login` и `supabase link` в каталоге репозитория).

### Проверки перед релизом

В корне репозитория:

| Команда | Зачем |
|--------|--------|
| `npm run supabase:migrations` | Локальные миграции совпадают с облаком |
| `npm run supabase:lint` | Схема на сервере без ошибок линтера |
| `npm run supabase:push` | Применить **новые** SQL-миграции из `supabase/migrations/` к облаку |

Сейчас база **на месте**: все миграции применены, `db lint --linked` без ошибок.

### OAuth и URL (обязательно для Google-входа)

1. В **Supabase Dashboard** → **Authentication** → **URL Configuration**:
   - **Site URL** — публичный адрес фронта (например `https://твой-сервис.up.railway.app/` или свой домен **со слешем в конце**, если так принято в настройках).
   - **Redirect URLs** — добавь **точно** тот же корень, что в коде после входа: `https://твой-домен/` (как в `AuthModal`: редирект на корень SPA).
2. В **Google Cloud Console** (OAuth-клиент для этого проекта) в **Authorized redirect URIs** должна быть ссылка вида  
   `https://<project-ref>.supabase.co/auth/v1/callback` (её даёт Supabase в настройках провайдера).

Не выполняй `supabase config push`, пока в `supabase/config.toml` в секции `[auth]` стоят **локальные** `127.0.0.1` — иначе можно затереть боевые URL в облаке. Для прода удобнее править **Site URL / Redirect URLs в Dashboard**; `config push` используй только осознанно, с прод-URL в `site_url` и в `additional_redirect_urls`.

### Локальный Supabase (Docker)

`supabase start` поднимает Postgres и Auth в Docker — нужен запущенный Docker Desktop. Для работы приложения с **облачным** проектом достаточно переменных `VITE_SUPABASE_*` в `.env`, локальный стек не обязателен.
