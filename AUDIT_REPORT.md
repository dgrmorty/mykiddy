# Аудит веб-приложения MyKiddy — детская IT-школа

**Дата:** 3 марта 2025  
**Роль:** Senior Software Architect, Security Auditor, UX-эксперт, Product-консультант  
**Цель:** Строгий аудит с конкретными рекомендациями до уровня топовой EdTech-платформы.

---

## Краткий общий вердикт

**Оценка: 5.5/10 — рабочий MVP с серьёзными рисками для продакшена и детской аудитории.**

Приложение выполняет базовый сценарий (курсы, уроки, ДЗ, ИИ-тьютор, админка), но имеет критические пробелы в безопасности, масштабируемости и защите персональных данных детей. Архитектура монолитная, без резервирования и rate limiting; секреты и конфигурация захардкожены; часть логики (прогресс на дашборде, RPC XP) может работать некорректно. Для реального запуска с детьми нужны срочные исправления в безопасности и надёжности, затем — планомерное усиление архитектуры и UX.

# 1. АРХИТЕКТУРА

## 1.1 Масштабируемость

| Проблема | Последствия | Решение |
|----------|-------------|---------|
| Один инстанс Express без горизонтального масштабирования | При росте нагрузки один процесс станет узким местом; отказ одного деплоя = полный даунтайм | Вынести статику на CDN; завести несколько инстансов за балансировщиком; сделать сервер stateless (сессии только в Supabase) |
| Кэш курсов в памяти (`contentService`: `coursesCache`) на клиенте и отсутствие кэша на сервере | При 10x пользователях каждый клиент дёргает Supabase; сервер не снижает нагрузку на БД | Серверный кэш (Redis/in-memory) для каталога курсов с TTL; опционально CDN для статики и API ответов |
| Нет разделения read/write, все запросы к одной Supabase | Рост нагрузки на одну БД; сложнее репликация и отказоустойчивость | Read replicas в Supabase (если план позволяет); разделить тяжёлое чтение (каталог, лидерборд) и запись |
| Монолит: SPA + API в одном репозитории и деплое | Любое изменение требует полного редеплоя; сложнее независимое масштабирование фронта и бэкенда | Разнести фронт (Vercel/Netlify/CDN) и бэкенд (Railway/другой хостинг); версионирование API |

## 1.2 Устойчивость к росту нагрузки

| Проблема | Последствия | Решение |
|----------|-------------|---------|
| Нет rate limiting на API | Один пользователь или бот может исчерпать квоту ИИ и создать нагрузку на Gemini | `express-rate-limit` на `/api/*` (например 100 req/min на IP + жёсткие лимиты на `/api/ai-tutor`, `/api/check-homework`) |
| Синхронные вызовы Gemini в обработчиках запросов | Долгий ответ ИИ блокирует поток; при всплеске запросов — таймауты и 503 | Очередь (Bull/Redis) + воркеры для ИИ; ответ клиенту через polling или WebSocket; таймауты на уровне reverse proxy |
| Один глобальный экземпляр `GoogleGenAI` в `server.js` | Нет изоляции по запросам; при падении инициализации все запросы к ИИ падают | Фабрика/пул или ленивая инициализация с проверкой здоровья; circuit breaker при повторяющихся ошибках Gemini |

## 1.3 Антипаттерны

| Антипаттерн | Где | Решение |
|-------------|-----|---------|
| Жёстко заданный production URL в коде | `config.ts`: `API_BASE_URL = 'https://mykiddy-production.up.railway.app'` | Использовать `import.meta.env.VITE_API_URL` с дефолтом; разные конфиги для dev/stage/prod |
| Секреты Supabase в клиентском коде | `services/supabase.ts`: URL и anon key захардкожены | Вынести в `VITE_SUPABASE_URL` и `VITE_SUPABASE_ANON_KEY`; anon key допустим в клиенте, но не смешивать с серверными секретами в одном файле |
| Fallback admin-email в коде | `AuthContext.tsx`: `['knazar002@gmail.com']` если нет `VITE_ADMIN_EMAILS` | Убрать дефолт; требовать явную настройку в env в продакшене |
| Роль админа по email в клиенте | Проверка `ADMIN_EMAILS.includes(email)` на фронте | Дублировать проверку на бэкенде для критичных действий; роль хранить в БД и доверять только серверу/RLS |

## 1.4 Узкие места и стратегия улучшения

- **Узкие места:** (1) Единственный процесс Node для API и раздачи SPA. (2) Прямые вызовы Gemini без очереди. (3) Множественные запросы к Supabase из админки (курсы → модули → уроки) без батчинга.
- **Стратегия:** Ввести rate limiting и таймауты; вынести статику на CDN; добавить серверный кэш каталога; поставить очередь для ИИ; документировать и при необходимости формализовать RLS и политики Supabase.

# 2. БЕЗОПАСНОСТЬ

## 2.1 Аутентификация и авторизация

| Проблема | Последствия | Решение |
|----------|-------------|---------|
| Список админов только по email и на клиенте | Подмена JWT или манипуляция данными может дать доступ к админ-функциям, если на бэкенде нет проверки роли | Хранить роль/флаг админа в `profiles`; на сервере для чувствительных операций проверять роль через Supabase (RLS или server-side); не полагаться на клиентский `isAdmin` |
| Нет проверки JWT на сервере для части операций | `/api/health` и при отсутствии Supabase `/api/ai-usage` не требуют токена — нормально; но все мутирующие и персональные эндпоинты должны проверять JWT | Явно проверять токен через `supabase.auth.getUser(accessToken)` для `/api/ai-tutor`, `/api/check-homework`, `/api/ai-usage` при наличии Supabase; при невалидном токене возвращать 401 |
| Регистрация без подтверждения email | Дети или третьи лица могут создавать аккаунты на произвольные email | Включить в Supabase "Confirm email"; показывать сообщение «Проверьте почту» после регистрации; опционально капча/ограничение по доменам для школ |
| Пароль только «required» и минимум Supabase (6 символов) | Слабые пароли повышают риск взлома аккаунтов | Клиентская валидация: минимум 8 символов, комбинация букв/цифр; рекомендовать родителям сложный пароль; не логировать и не передавать пароль нигде кроме Supabase Auth |

## 2.2 Защита данных и детская аудитория

| Проблема | Последствия | Решение |
|----------|-------------|---------|
| Нет явной политики и технических мер под GDPR/детские данные | Юридические и репутационные риски; данные детей требуют повышенной защиты | Документировать сбор данных; минимизировать данные (не хранить лишнего); согласие родителей для регистрации детей; возможность экспорта и удаления данных |
| ИИ получает текст ребёнка (вопросы, ответы ДЗ) | Утечка в логи сторонних API (Gemini); использование данных не по назначению | В условиях использования указать передачу в провайдера ИИ; по возможности отключить логирование в Google AI; не отправлять в ИИ личные данные (ФИО, контакты) |
| Публичный лидерборд с именами и email | Раскрытие персональных данных детей | Лидерборд по никам/аватарам без email; настройка «показывать меня в рейтинге»; RLS на `profiles` так, чтобы не отдавать email для списка рейтинга |
| Хранение email в `profiles` и отображение в админке | Утечка email всех пользователей при компрометации админки | Строгий RLS: только админы видят email; логировать доступ к списку пользователей; 2FA для админов |

## 2.3 Типичные атаки

| Угроза | Текущее состояние | Рекомендация |
|--------|-------------------|--------------|
| XSS | Частичная санитизация в `utils/security.ts`; React экранирует по умолчанию | Сохранить санитизацию для вставок в ИИ и ДЗ; не рендерить HTML из пользовательского ввода без DOMPurify; заголовок Content-Security-Policy |
| CSRF | API без cookie-сессий, Bearer в заголовке — риск ниже | Для форм, если появятся cookie-сессии, ввести CSRF-токены; сейчас придерживаться stateless JWT |
| Инъекции в ИИ (prompt injection) | Есть `isPotentialInjection` и базовая фильтрация | Расширить список паттернов; на сервере дублировать проверку; ограничить длину и содержание контекста в промпте |
| Брутфорс логина | Нет rate limit на Supabase Auth | Ограничение на стороне Supabase (если есть) или проксирование входа через свой бэкенд с rate limit |
| Перебор и злоупотребление API | Нет rate limit на Express | Ввести `express-rate-limit` и отдельные лимиты на тяжёлые эндпоинты |

## 2.4 Конкретные усиления

- Включить **Helmet** на Express (безопасные заголовки).
- Добавить **Content-Security-Policy** в `index.html` или через Helmet (ограничить источники скриптов, стилей, фреймов).
- Реализовать **rate limiting** на уровне приложения и при необходимости на уровне nginx/Railway.
- Убедиться, что **RLS в Supabase** настроен для всех таблиц (`profiles`, `user_progress`, `homework_submissions`, `ai_usage`, `courses`, `modules`, `lessons`, `schedule_events`, `settings`): пользователь видит только свои данные, админ — по роли.
- Не хранить и не логировать пароли; не передавать в логи токены и персональные данные детей.

# 3. СЕТЕВОЕ ВЗАИМОДЕЙСТВИЕ

## 3.1 Корректность запросов

| Проблема | Последствия | Решение |
|----------|-------------|---------|
| Жёсткий API_BASE_URL в клиенте | Смена домена/окружения требует пересборки | `VITE_API_URL` с fallback на относительный путь для того же origin |
| Нет единого API-клиента (fetch разбросан по сервисам) | Дублирование заголовков, разное поведение при ошибках | Общий `apiClient(url, { method, body, token })` с обработкой 401/429/5xx и таймаутами |
| В `checkHomework` нет retry при сетевой ошибке | В отличие от `askAiTutor` один сбой = сразу ошибка пользователю | Один повтор при сетевой/временной ошибке с экспоненциальной задержкой |

## 3.2 Обработка ошибок

| Проблема | Последствия | Решение |
|----------|-------------|---------|
| При таймауте БД в `contentService` возвращается пустой массив | Пользователь видит «Курсы не добавлены» вместо «Временная ошибка» | Различать таймаут и пустой ответ; показывать сообщение «Не удалось загрузить. Повторите позже» и кнопку повтора |
| В `server.js` при ошибке Gemini в ответ уходит `error.message` | Раскрытие внутренних сообщений клиенту | Возвращать общий текст «Сервис временно недоступен»; детали писать только в серверный лог |
| Нет структурированного кода ошибок в API | Клиенту сложно различать «нет сети», «лимит», «не авторизован» | Стандартный формат `{ error: string, code?: string }`; коды типа `AUTH_REQUIRED`, `AI_LIMIT`, `TIMEOUT` для точечной обработки на UI |

## 3.3 Устойчивость к плохому интернету

| Проблема | Последствия | Решение |
|----------|-------------|---------|
| Таймаут 3 с для загрузки курсов и профиля | На медленной сети частые «пустые» экраны | Увеличить до 8–10 с для каталога; показывать скелетон/прогресс; повтор по кнопке |
| Один таймаут на весь запрос к ИИ (90/60 с) | Долгое ожидание без обратной связи | Показывать «ИИ думает…» и возможность отмены; опционально streaming ответа от Gemini для появления текста по мере генерации |
| Нет индикации «офлайн» | Пользователь не понимает, почему не грузится | Использовать `navigator.onLine` и событие `online`/`offline`; баннер «Нет соединения» и отключение отправки форм |

## 3.4 Таймауты и повторы

| Рекомендация | Детали |
|--------------|--------|
| Таймауты на сервере | Для вызовов Gemini задать таймаут (например 60 с) и при превышении отвечать 504 с кодом `AI_TIMEOUT` |
| Повторы на клиенте | Для идемпотентных GET и для ИИ — один повтор с задержкой 2–3 с при сетевой ошибке/таймауте |
| AbortController | Уже используется в `geminiService` — сохранить; при уходе со страницы отменять запросы |

# 4. БАЗА ДАННЫХ

## 4.1 Структура таблиц (по коду)

- **profiles** — пользователи (id, email, name, role, avatar, level, xp, is_approved и т.д.).
- **courses**, **modules**, **lessons** — иерархия контента.
- **user_progress** — пары (user_id, lesson_id) для пройденных уроков.
- **homework_submissions** — сдача ДЗ (user_id, lesson_id, xp_awarded).
- **ai_usage** — суточные счётчики (user_id, date, tutor_count, homework_count).
- **schedule_events**, **settings** — расписание и настройки.

Миграции в репозитории не найдены — структура задаётся в Supabase. Ниже — типичные риски.

## 4.2 Нормализация и целостность

| Проблема | Последствия | Решение |
|----------|-------------|---------|
| Удаление курса/модуля без каскада | Сироты в `lessons`, `user_progress`, `homework_submissions` | ON DELETE CASCADE для FK (lesson → module → course); либо мягкое удаление (deleted_at) |
| Дублирование прогресса и XP | `markLessonComplete` + `increment_xp` и отдельно запись в `homework_submissions` с xp_awarded | Единый источник истины: XP только через RPC `increment_xp` с привязкой к user; в RPC обновлять и profiles.xp, и при необходимости историю |
| RPC `increment_xp` вызывается без явного user_id | В Supabase RPC обычно получает user из JWT; если RPC не использует auth.uid() — XP может начисляться не тому пользователю | Проверить в БД: `increment_xp` должен использовать auth.uid(); в коде не передаётся user_id — допустимо только при гарантии контекста JWT |

## 4.3 Индексы и производительность

| Таблица / запрос | Рекомендуемые индексы |
|------------------|------------------------|
| user_progress | (user_id), (lesson_id), уникальный (user_id, lesson_id) |
| homework_submissions | (user_id, lesson_id) для проверки «уже сдано» |
| ai_usage | Уникальный (user_id, date) для upsert и выборки по дню |
| profiles (лидерборд) | (xp DESC) или составной для сортировки по level/xp |
| lessons | (module_id); modules (course_id) |

Без индексов запросы при росте данных станут медленными.

## 4.4 Скорость запросов и оптимизация

| Проблема | Решение |
|----------|---------|
| В админке N+1: курсы → модули → уроки отдельными запросами | Один запрос с вложенным select: `courses → modules → lessons` (как в contentService); кэшировать на клиенте при переключении вкладок |
| Большая выборка курсов с вложенными модулями/уроками | Пагинация или ленивая подгрузка модулей; на главной показывать только первый курс / следующий урок |
| Частые запросы к `profiles` при смене вкладки | Троттлинг уже есть (3 мин); можно увеличить TTL или использовать только auth.getSession() без перезапроса профиля |

# 5. ПРОИЗВОДИТЕЛЬНОСТЬ

## 5.1 Утечки памяти

| Риск | Где | Решение |
|------|-----|---------|
| Подписки и таймеры при размонтировании | AuthContext: `onAuthStateChange`, `visibilitychange` — отписка есть; в CourseDetail таймер 2.2 с при переходе в урок | Убедиться, что при размонтировании CourseDetail отменяются все таймеры и запросы (AbortController для fetch в loadData) |
| Глобальный кэш курсов без ограничения размера | `coursesCache` один объект, перезаписывается — риск низкий | При появлении нескольких ключей кэша (например по ролям) ввести максимум записей или TTL по давности |
| Recharts и тяжёлые компоненты на дашборде | RadarChart на каждом рендере | Оставить как есть; при росте данных — виртуализация или ленивая загрузка графика |

## 5.2 Тяжёлые операции

| Проблема | Решение |
|----------|---------|
| Сортировка модулей/уроков на клиенте после каждого ответа БД | Делать order в Supabase; на клиенте не сортировать заново или один раз при маппинге |
| Много ре-рендеров из-за контекстов | Разбить AuthContext (user vs loading vs actions); мемоизация тяжёлых списков (React.memo, useMemo для списка курсов в CourseDetail) |
| Анимация перехода в урок 2.2 с блокировкой UI | Уменьшить до 1–1.5 с или сделать прелоад следующего контента во время анимации |

## 5.3 Загрузка видео

| Проблема | Решение |
|----------|---------|
| YouTube iframe без lazy load | Добавить loading="lazy" или загрузку iframe только при появлении в viewport (IntersectionObserver) |
| Прямые ссылки на video/mp4 без адаптивного потока | Для своих видео рассмотреть HLS/DASH или хотя бы несколько качеств; указать preload="metadata" по умолчанию |
| Нет индикации загрузки видео | Показывать спиннер поверх плеера до события canplay |

## 5.4 Первый рендер

| Проблема | Решение |
|----------|---------|
| Шрифты с Google Fonts блокируют рендер | preconnect есть; добавить font-display: swap в запрос шрифтов |
| Tailwind с CDN | Для продакшена перейти на сборку Tailwind в Vite — меньше объём и быстрее загрузка |
| Критичный CSS не инлайнится | Основные стили выше-the-fold вынести в inline в index.html для ускорения первого отображения |

# 6. UX/UI

## 6.1 Удобство для детей

| Проблема | Последствия | Решение |
|----------|-------------|---------|
| Много текста и мелкие подписи (uppercase, 10px) | Сложно читать младшим школьникам | Увеличить размер шрифта для ключевых подписей; основные действия — крупные кнопки с иконками и коротким текстом |
| Длинные сообщения об ошибках | Ребёнок не понимает, что делать | Короткие формулировки: «Что-то пошло не так. Нажми „Повторить“»; основная кнопка — повтор действия |
| Переключение роли Student/Parent без пояснения | Неочевидно для ребёнка и родителя | Подпись «Режим ученика» / «Режим родителя» и краткое объяснение при первом переключении |

## 6.2 Читаемость и навигация

| Проблема | Решение |
|----------|---------|
| Нет breadcrumbs в курсе (Курс → Модуль → Урок) | Добавить цепочку навигации над заголовком урока |
| Модальное окно курса с длинным списком уроков | Список уроков с прокруткой и явным «Закрыть»; на мобильном — полноэкранный список вместо мелкого модала |
| Два разных места для «Завершить урок» и «Сдать ДЗ» | Оставить как есть, но визуально связать блок «Практика» с кнопкой «Сдать работу» и состоянием «Задание выполнено» |
| События на дашборде захардкожены | Подтягивать из `schedule_events`; если пусто — показывать «Скоро появятся события» или скрывать блок |

## 6.3 Риски путаницы

| Ситуация | Решение |
|----------|---------|
| После сдачи ДЗ кнопка «Проверить» остаётся активной до обновления | После успешной проверки сразу ставить `isHomeworkCompleted` и обновлять кэш/состояние; не полагаться только на refetch |
| Гость видит «Продолжить обучение» без контекста | Текст «Войдите, чтобы сохранять прогресс» и явная кнопка входа |
| «Профиль навыков» с фиктивными данными (SKILL_DATA) | Либо привязать к реальным метрикам (по курсам/урокам), либо переименовать в «Пример навыков» или убрать до появления данных |

# 7. ЛОГИКА ПРОДУКТА

## 7.1 Ценность и лишние элементы

| Элемент | Оценка | Рекомендация |
|---------|--------|--------------|
| ИИ-тьютор с лимитом запросов | Высокая ценность | Сохранить; показывать оставшиеся запросы и «лимит обновится завтра» |
| Проверка ДЗ через ИИ | Высокая ценность | Сохранить; добавить короткую обратную связь «Молодец!» / «Попробуй ещё раз» до полного ответа |
| Радар «навыков» на дашборде с константой SKILL_DATA | Низкая ценность, вводит в заблуждение | Убрать или заменить на реальную статистику (пройдено уроков по курсам) |
| Захардкоженные события на дашборде | Ложная информация | Показывать только реальные события из БД |
| Переключение Student/Parent без разного контента | Неочевидно | Либо дать разный вид (родитель: прогресс ребёнка, расписание), либо убрать переключатель до появления функционала |

## 7.2 Чего не хватает

- **Восстановление пароля** — ссылка «Забыли пароль?» в AuthModal и поток через Supabase.
- **Ограничение доступа по возрасту / согласие родителя** — явный чекбокс и хранение факта согласия.
- **Уведомления** — напоминания о следующем уроке или дедлайне ДЗ (email/push при появлении бэкенда).
- **Прогресс по курсу в процентах** — уже есть в карточке курса; на дашборде показывать «Ты прошёл X% курса».
- **Сертификат или награда** за прохождение курса — повышает вовлечённость.

## 7.3 Вовлечённость и мотивация детей

- Геймификация: уже есть XP и уровень — добавить бейджи/достижения за серии уроков, за первое ДЗ, за N запросов к ИИ.
- Ежедневная серия (streak): «Ты заходил 3 дня подряд!».
- Короткие позитивные сообщения после завершения урока и после успешной сдачи ДЗ.
- Лидерборд — оставить, но без персональных данных (никнейм/аватар), с опцией скрыться.

# 8. КОДОВАЯ БАЗА

## 8.1 Организация файлов

| Проблема | Решение |
|----------|---------|
| В корне смешаны конфиг, константы, типы, сервер | Папки `src/` (или `app/`), `server/`, `shared/`; конфиг и типы в отдельных модулях |
| Крупные компоненты (AdminPanel, CourseDetail) | Разбить на подкомпоненты: AdminPanel → ContentView, UsersView, ScheduleView; CourseDetail → LessonPlayer, HomeworkCard, CourseList |
| Нет папки api/ на клиенте | Вынести вызовы API в `api/` или оставить в `services/` с чётким разделением: auth, content, ai |

## 8.2 Дублирование логики

| Дублирование | Решение |
|--------------|---------|
| Таймаут через Promise.race с setTimeout в AuthContext и contentService | Общая утилита `withTimeout(promise, ms)` |
| Повторяющаяся разметка карточек курсов/уроков | Переиспользуемый компонент CardCourse, CardLesson с пропсами |
| Проверка isGuest и requireAuth в нескольких экранах | HOC или wrapper `withAuth(Component)` для защищённых экранов; единая точка редиректа/модалки входа |

## 8.3 Сложные и хрупкие места

| Место | Проблема | Решение |
|-------|----------|---------|
| CourseDetail: состояние урока, ДЗ, модалки, анимация в одном компоненте | Трудно тестировать и менять | Вынести состояние ДЗ в хук useHomework(lessonId, userId); состояние плеера и модалки курса — в отдельные компоненты |
| finalizeHomeworkReward: порядок вызовов (markLessonComplete, increment_xp, homework_submissions, refreshUser) | При сбое на середине — несогласованность | Одна серверная RPC «complete_homework(lesson_id)» или транзакция на бэкенде: обновление прогресса + XP + запись в homework_submissions |
| AdminPanel: множество useState и обработчиков в одном файле | Высокий когнитивный груз | Разбить на вьюхи; формы в отдельные компоненты с локальным состоянием; общее состояние (список курсов, пользователей) — контекст или серверный кэш |

# 9. ОБРАБОТКА ОШИБОК

## 9.1 Поведение при сбоях

| Сценарий | Текущее поведение | Рекомендация |
|----------|-------------------|--------------|
| Supabase недоступен при инициализации | Таймаут 5 с → гость | После таймаута показывать баннер «Проблемы с подключением» и кнопку «Повторить» |
| Ошибка при загрузке курсов | Пустой список, кнопка «Переподключиться» в CourseDetail | Единый паттерн: сообщение + повтор; в useContent передавать userId и при ошибке не подменять пустым массивом без пояснения |
| Ошибка при отправке ДЗ | Сообщение в setAiFeedback | Сохранить; добавить кнопку «Отправить снова» без потери введённого текста |
| Падение React (ErrorBoundary) | Fallback с «Обновите страницу» | Добавить кнопку «Обновить» и опционально отправку отчёта об ошибке (без персональных данных) |

## 9.2 Сообщения пользователю

- Избегать технических терминов («401», «CORS», «DATABASE_TIMEOUT»).
- Короткие формулировки: «Не удалось загрузить. Попробуйте ещё раз.» / «Сервис временно недоступен.» / «Войдите в аккаунт.»
- Для детей: позитивный тон, одна основная кнопка действия.

## 9.3 Где приложение может «ломаться»

- **Отсутствие userId в useContent**: дашборд и AiTutor получают курсы без прогресса (все уроки «не пройдены»). Нужно передавать user.id в useContent и в getCourses.
- **RPC increment_xp**: если в БД RPC не привязан к auth.uid(), XP может начисляться не тому пользователю — проверить в Supabase.
- **Двойное начисление за ДЗ**: при быстром двойном клике «Проверить» возможны два запроса — на сервере и в БД делать идемпотентность (например по паре user_id + lesson_id для проверки ДЗ).
- **Админка**: при отключении RPC get_all_users показываются только профили из таблицы profiles; пользователи только что зарегистрированные могут не попасть в список — документировать и при необходимости джойн с auth.users через серверную функцию.

# 10. РИСКИ И МАСШТАБИРОВАНИЕ

## 10.1 Что сломается первым при 10x росте

1. **Исчерпание квоты/лимитов Gemini** — нужна очередь, приоритизация, возможно кэш типовых ответов.
2. **Нагрузка на один инстанс Express** — несколько воркеров или инстансов за балансировщиком.
3. **Supabase connection pool и лимиты запросов** — оптимизация запросов, кэш, read replicas.
4. **Жёсткие лимиты в коде** (например лидерборд 50) — вынести в конфиг; пагинация.

## 10.2 Что усиливать заранее

- Rate limiting и Helmet на API.
- Проверка роли админа на бэкенде для любых чувствительных операций.
- RLS и индексы в Supabase.
- Переменные окружения для всех URL и секретов; убрать дефолтный admin email из кода.
- Единый обработчик ошибок API и коды ошибок для клиента.

## 10.3 Подготовка к росту школы

- Мониторинг: здоровье API (/api/health), метрики запросов, ошибки (Sentry или аналог).
- Логирование без персональных данных; ротация логов.
- Резервное копирование БД и план восстановления.
- Документация по развёртыванию и env; отдельные окружения stage/prod.
- По мере роста: отдельный бэкенд для ИИ (очередь, воркеры), CDN для статики и медиа, реплики БД.

# ПРИОРИТЕТЫ ВНЕДРЕНИЯ

**Срочно (до запуска с детьми):**
1. Безопасность: Helmet, CSP, rate limiting; проверка роли админа на бэкенде; RLS и политики для детских данных.
2. Исправление багов: useContent(userId) для корректного прогресса на дашборде; проверка RPC increment_xp в БД.
3. Секреты и конфиг: вынести URL и ключи в env; убрать fallback admin email.

**Краткосрочно:**
4. Обработка ошибок и сообщения пользователю; восстановление пароля; подтверждение email.
5. Оптимизация БД: индексы, один запрос для админки вместо N+1.
6. UX: читаемость для детей, реальные события на дашборде, убрать или заменить фейковый радар навыков.

**Среднесрочно:**
7. Очередь для ИИ, таймауты на сервере, CDN для статики.

9. Рефакторинг крупных компонентов и общего API-клиента.
