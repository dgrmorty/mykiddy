# Слабые места проекта

## Безопасность

1. **Админка только на фронте**  
   Роль админа задаётся по email в `AuthContext.tsx` (и при необходимости через `VITE_ADMIN_EMAILS`). Доступ к админ-странице закрыт маршрутом, но запросы к Supabase (удаление пользователей, правка курсов, расписания) идут с клиента. **Критично:** в Supabase для таблиц `courses`, `modules`, `lessons`, `schedule_events`, `settings` и для RPC `delete_user_by_admin`, `get_all_users` должны быть настроены RLS-политики так, чтобы эти операции были разрешены только админам (например, по `auth.jwt() ->> 'email'` или по роли в `profiles`). Иначе любой авторизованный пользователь может вызвать те же методы из консоли браузера.

2. **Секреты и ключи в коде**  
   В `services/supabase.ts` захардкожены URL и anon-ключ Supabase. Anon-ключ рассчитан на публичное использование, но при смене проекта их нужно менять в коде. Для смены окружений лучше вынести в `import.meta.env.VITE_SUPABASE_URL` и `VITE_SUPABASE_ANON_KEY`.

3. **API_KEY только на сервере**  
   Ключ Gemini хранится в переменных Railway (`API_KEY`) — корректно, в коде не светится.

## Надёжность и прод

4. **Нет общего rate limit на API**  
   Лимиты есть только для ИИ (тьютор и проверка ДЗ). Остальные эндпоинты (`/api/health`, `/api/ai-usage`) можно ддосить. Имеет смысл добавить общий лимит запросов по IP или по пользователю (например, через middleware).

5. **Ошибки и логи**  
   В коде много `console.log` / `console.error`. В проде лучше слать логи в внешний сервис и не выводить детали ошибок пользователю (уже в целом ок, но стоит проверить, что в ответах API нет стектрейсов и внутренних сообщений).

6. **Роль админа**  
   Список админов задаётся через `VITE_ADMIN_EMAILS` (через запятую в .env). Если переменная не задана, используется fallback-email в коде — для публичного репозитория лучше всегда задавать `VITE_ADMIN_EMAILS` при сборке и убрать fallback.

## Архитектура и код

7. **Роутинг только на клиенте**  
   Используется `HashRouter`. Для SEO и красивых URL можно перейти на `BrowserRouter` и настроить сервер (Railway) на отдачу `index.html` для всех маршрутов (уже сделано в `server.js` для прода).

8. **Один конфиг для всех окружений**  
   `config.ts` с фиксированным `API_BASE_URL`. Для dev/staging/prod удобнее использовать `VITE_API_BASE_URL` и при сборке подставлять нужный URL.

9. **Зависимость от Supabase RLS**  
   Файлы `.sql` удалены; схема и RLS остаются только в БД. При новом проекте Supabase нужно заново создать таблицы и политики (миграции хранить отдельно или восстановить из бэкапа).

## Что уже сделано хорошо

- Санитизация ввода и проверка на prompt injection в чате и при проверке ДЗ.
- CORS ограничен доменами и Railway.
- Лимиты запросов к ИИ по JWT и по дням.
- Пароли и ключи не попадают в ответы API; токен передаётся в заголовке и проверяется на сервере.
